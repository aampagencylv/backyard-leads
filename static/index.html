Developer

Push data into Prospector via REST. Receive event notifications via signed webhooks. Set up keys + webhook subscriptions in Settings → 🔑 API Keys & Webhooks.

Quick Start (3 steps)

1. Generate an API key

Open Settings → 🔑 API Keys & Webhooks → + New Key. Save the plaintext (pk_live_...) — it's shown once, then SHA-256-hashed at rest.

2. Push a company in

curl -X POST https://prospector.backyardmarketingpros.com/api/v1/companies \
  -H "X-API-Key: pk_live_YOUR_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Smith Pools",
    "website": "smithpools.com",
    "phone": "+17025551234",
    "city": "Las Vegas",
    "state": "NV",
    "business_type": "pool builders"
  }'

Response includes created: true|false — duplicate domains upsert into the existing record without overwriting populated fields.

3. Subscribe to events (webhooks)

Settings → + New Webhook. Paste your endpoint URL, check email.replied + meeting.booked (or any subset), save the signing secret. We'll POST every matching event with HMAC-SHA256 in X-Webhook-Signature.

Test the wiring without real data

Use the Test button on each webhook in Settings to fire a synthetic webhook.test event. Use webhook.site or ngrok during development.

Authentication

Every /api/v1/* request requires the X-API-Key header.

X-API-Key: pk_live_YOUR_KEY

Key format

pk_live_ + 64 hex characters (32 random bytes). Generated in Settings, shown once at creation, SHA-256-hashed at rest. We never store the plaintext.

Scoping

Keys inherit their owner's role + scoping:

  • sales_rep keys see only the rep's assigned companies/contacts/deals
  • admin / super_admin keys see everything in the org

Revocation

Revoke any key from Settings — soft-delete (preserves audit trail of when the key was used). Revoked keys 401 immediately on the next request.

Rate limits

v1 is uncapped — we'll add per-key throttling if any single key starts hammering the API. Until then, please be reasonable: a few req/sec is fine, sustained 100+ req/sec will get noticed.

Webhooks

Outbound HTTP POSTs to your endpoint when subscribed events fire. Signed with HMAC-SHA256 so you can verify requests came from us.

Setup

  1. Settings → + New Webhook → paste your URL, check the events you want
  2. Copy the signing secret (shown once) into your endpoint config
  3. Verify signatures on every request

Headers we send

Content-Type:        application/json
X-Webhook-Event:     company.created
X-Webhook-Signature: sha256=<hex_hmac_of_raw_body>
X-Webhook-Delivery:  <random_token>
User-Agent:          BMP-Prospector-Webhooks/1.0

X-Webhook-Delivery is unique per attempt — store it on your end + dedupe so retries don't double-process.

Verify the signature (Python)

import hmac, hashlib

def verify(secret: str, raw_body: bytes, header: str) -> bool:
    expected = "sha256=" + hmac.new(
        secret.encode(), raw_body, hashlib.sha256
    ).hexdigest()
    return hmac.compare_digest(expected, header)

# In your endpoint:
sig = request.headers["X-Webhook-Signature"]
if not verify(MY_WEBHOOK_SECRET, request.body_raw, sig):
    return 401, "bad signature"

Verify the signature (Node.js)

const crypto = require('crypto');

function verify(secret, rawBody, header) {
  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(rawBody)
    .digest('hex');
  return crypto.timingSafeEqual(
    Buffer.from(expected), Buffer.from(header)
  );
}

Body shape (every event)

{
  "event": "company.created",
  "delivery_id": "abc123_unique_per_attempt",
  "occurred_at": "2026-05-08T19:42:11+00:00",
  "data": { ... event-specific payload ... }
}

Delivery + retry

v1 is best-effort: 10s timeout, single attempt. Failed deliveries are visible in Settings → Webhooks (last status, last error, failure count). Retry queue lands in v2.

Respond with any 2xx to acknowledge. 4xx / 5xx / network errors bump the failure counter; we don't auto-disable.

Test without real events

Settings → Webhooks → Test (per webhook). Fires a synthetic webhook.test event with a token + your email. Use it to verify signature handling end-to-end.

Errors & Limits

Error format

FastAPI standard:

{ "detail": "Invalid or revoked API key" }

Status codes

CodeMeaning
200Success on GET / PATCH
201Success on POST creating a new record
400Validation error (missing required fields, bad URL, unknown event name, etc.)
401Missing or invalid X-API-Key header, or revoked key
403Authenticated but the resource is outside your scoping
404Record not found OR exists but you don't have permission
500Internal — never your fault, ping us if you see one

Limits (v1)

  • Pagination: limit max 200 per request, offset-based
  • Rate limit: none enforced; please be reasonable. We'll add per-key throttling if abuse appears.
  • Webhook timeout: 10s per delivery attempt
  • Custom fields: only keys matching active definitions are accepted; unknown keys silently dropped
  • Body size: no enforced limit, but POSTs larger than 1MB will be rejected by nginx

Coming in v2

  • Webhook retry queue (currently single-attempt)
  • Per-key rate limiting
  • Cursor-based pagination
  • More events: sequence.completed, deal.stage_changed, company.updated
  • Native Zapier app (currently you can use Zapier's generic Webhooks-by-Zapier integration)
